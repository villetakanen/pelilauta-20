# Feature: Cyan Living Style Book (app/cyan-ds)

## Blueprint

### Context
The Living Style Book (`app/cyan-ds`) is the source of truth for the Cyan Design System. It provides a sterile environment to develop, document, and verify components and tokens in isolation before they are integrated into the main RPG platform (`app/pelilauta`). It serves as the primary host for visual regression and E2E behavioral tests.

### Architecture
- **Framework:** Astro (SSR-first).
- **Core Architecture:** **Astro Content Collections** for all "books" (documentation entries), separating them from structural pages.
- **Registries (Collections):** Documentation is strictly categorized into four groups:
    - `principles`: Core identity and branding foundations (Chroma, Semantic Colors, Thematic Identity).
    - `styles`: CSS and structural tokens (Grid, Units, Z-index, Elevation).
    - `components`: Core Design System components (shared by all apps).
    - `addons`: Specialized micro-frontend / sub-site components (MFE-only).
- **Routing:** All dynamic pages use a mandatory collection-prefixed route: `/[collection]/[slug]` (e.g., `/principles/chroma`, `/styles/units-and-grid`). This hierarchy ensures zero slug collisions and reflects the data organization.

- **Layered layouts:**
    - [`AppShell`](../layouts/app-shell/spec.md) — HTML shell (`<html>`, `<body>`, body grid, layout modes). Part of `packages/cyan`.
    - [`Page`](../layouts/page/spec.md) — Standard non-book page layout composed on `AppShell`. Part of `packages/cyan`. Does not own H1 or wrap content.
    - `app/cyan-ds/src/layouts/Book.astro` — Documentation-book layout composed on `Page`. Owns the primary page heading (H1) and optional `description` from frontmatter. By default wraps content in a prose `<article>`; opts out when `multipart: true` is set.
- **Site navigation:** Rendered by [`DocsTray`](../components/docs-tray/spec.md) — a site-wide component dropped into the `tray` slot of every page (Book-rendered or not). Book does NOT inline collection fetches or tray markup; it composes `DocsTray` via `Page`'s tray slot.
- **Book-authoring primitives:** DS-internal Astro components that MDX pages consume to honour the "No App-Layer Overrides" rule without reaching for inline styles or `<style>` blocks. Each primitive has its own spec under `specs/cyan-ds/living-style-books/`.
    - [`ThemeSplit`](theme-split/spec.md) — side-by-side light/dark pane wrapper for dual-theme demos.

### Data Models
- **Book Frontmatter Schema (Zod):**
    ```typescript
    {
      title: z.string(), // Page header
      description: z.string().optional(),
      icon: z.string().optional(),
      order: z.number().optional(), // Sorting priority within its category
      status: z.enum(['stable', 'alpha', 'draft']).default('stable'),
      multipart: z.boolean().optional().default(false) // Opt out of the prose <article> wrapper
    }
    ```

### Multipart Books

- By default, Book renders a `<header>` containing the page H1 and optional description.
- For prose documents (`multipart: false`), the header and MDX slot are wrapped in an `<article>` whose width is capped for readability (aligned with the `.cn-content-prose` contract — `min(67ch, 100%)` centered).
- MDX authors should begin body content with `## Heading` (H2); H1 is owned by Book, not the content.

When `multipart: true`:
- Book still renders the `<header>` with H1 + description.
- **Book does NOT wrap the MDX slot.** The slot is placed directly inside `<main class="cn-app-main">` so the MDX author owns its own container structure.
- The author is responsible for arranging their content into one or more `<section class="cn-content-prose">`, `<section class="cn-content-golden">`, `<section class="cn-content-triad">` shells. This unlocks main-wide demos (e.g. Golden and Triad layout demos) that cannot render correctly inside the single-prose cage.

### Dependencies
- **Depends on:** `packages/cyan` (AppShell, Page, Tray components, tokens), `packages/pelilauta-icons` (Tier 1 Assets).
- **Consumer:** Playwright E2E tests target dynamic routes generated by content collections.

### Anti-Patterns
- **No Tray Logic in Book.** Book must NOT call `getCollection` for navigation, build group structures, or render `TrayButton` / `TrayLink` markup. All navigation lives in [`DocsTray`](../components/docs-tray/spec.md).
- **No Direct Pages for Books:** Documentation entries must live in `src/content/[coll]/` rather than `src/pages/`.
- **No Redundant H1 in Content:** Markdown (`.mdx`) files must **not** include an H1 heading (e.g., `# Title`). The Book layout manages the primary page heading to avoid 3x title duplication (TopBar, Layout Header, and Content Start). This applies to both single-part and multipart books.
- **H2 for Content Sections:** Documentation content should start strictly with H2 (`## Heading`) to maintain correct hierarchical flow.
- **`multipart: true` is not a readability escape hatch.** Do not flip `multipart: true` on a single-part prose doc just to widen its content. Multipart is for pages whose body genuinely requires multiple top-level layout shells (e.g. Golden/Triad demos). When in doubt, keep `multipart: false`.
- **No App-Layer Overrides.** Book (and any Book-rendered MDX) MUST NOT contain `<style>` blocks or inline styles that override DS behavior. Missing DS capability is a DS bug, fixed in `packages/cyan` — not papered over in the book.

## Contract

### Definition of Done
- [ ] Documentation entries are stored in the correct `src/content/` subfolder.
- [ ] `src/content/config.ts` enforces the mandatory `title`, optional `order`, and optional `multipart` frontmatter.
- [ ] The site tray is rendered exclusively by [`DocsTray`](../components/docs-tray/spec.md). Book.astro contains no collection fetches or tray markup.
- [ ] Dynamic routing correctly renders pages at `/[coll]/[slug]`.
- [ ] `multipart: true` in a book's frontmatter causes Book to skip the prose `<article>` wrapper; the MDX slot is placed directly under `<main class="cn-app-main">`.
- [ ] Default (`multipart: false`) behavior wraps the MDX slot in a prose-capped `<article>` as before.

### Regression Guardrails
- **Schema Safety:** Build fails if any documentation entry violates the mandatory frontmatter contract.
- **Slug Collisions:** Unique slugs are enforced per category.
- **Layout Composition:** Book MUST compose on `Page`, not directly on `AppShell`. Page owns the shell; Book owns the book-specific adornments.

### Testing Scenarios

#### Scenario: Book renders single-part doc inside a prose article wrapper
```gherkin
Given a documentation entry without a multipart flag
When the entry is rendered
Then the MDX content is wrapped in an <article> whose effective width is capped at the prose contract (67ch centered)
  And the Book-owned <h1> appears once inside the article
```
- **Playwright E2E Test:** `app/cyan-ds/e2e/ds.spec.ts`

#### Scenario: Book renders multipart doc without the prose wrapper
```gherkin
Given a documentation entry with "multipart: true" in its frontmatter
When the entry is rendered
Then no <article max-width> wrapper is introduced by Book around the MDX slot
  And the MDX slot is a direct child of <main class="cn-app-main">
  And the Book-owned <h1> appears once above the slot
```
- **Playwright E2E Test:** `app/cyan-ds/e2e/ds.spec.ts`

#### Scenario: Documentation Navigation
```gherkin
Given the Cyan Living Style Book is running
When a user visits the root page
Then they see the DocsTray navigation with groups Principles, Styles, Components, and Addons
  And clicking a component link (e.g., "Tray") navigates to its documentation
```
- **Playwright E2E Test:** `app/cyan-ds/e2e/ds.spec.ts`

#### Scenario: Visual Regression Host
```gherkin
Given a component has a documented demo page
When visual regression tests are executed via Playwright
Then the demos on the page are evaluated for alignment, color, and state
  And the results match established snapshots
```
- **Playwright E2E Test:** `app/cyan-ds/e2e/components/[component].spec.ts`
