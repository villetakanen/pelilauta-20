Cyan treates color as a highly structured, scalable language. We utilize a **Layered Color Model** that separates mechanical brand identity from functional user intent.

> #### The 3-Tier Token Model
>
> We provide 3 tiers of tokens.
>
> 1.  **The Chroma Layer (Mechanical):** Raw tonal scales (OKLCH) where hand-tuned perceptual values live. This layer is **Swappable** for sub-site branding.
> 2.  **The Semantic Layer (Functional):** Maps Chroma steps to functional roles (e.g., `--color-text`, `--color-button-accent`). This layer is a **Stable API**.
> 3.  **The Component Layer (Consumptive):** Components only consume semantic tokens. They never reference raw HSL or hex values.

## Colors

Primary and surface colors are the foundation of the color system. They are used to create a consistent and predictable visual language that is easy to understand and use. The primary and surface colors are defined in a 13-point tonal scale following the **Material 3 (M3)** philosophy, where each color has a specific lightness value.

In addition to the primary and surface colors, there are also functional palettes, for error, warning, info and reactions. For the functional palettes, only a subset of the tonal scale is provided

The tonal step number (0-100) represents **perceived lightness**:
- **0** corresponds to absolute black.
- **100** corresponds to absolute white.
- Any intermediate number (e.g., **40**) represents a percentage of perceptual difference from black.

This creates a consistent relationship between steps across all hues, ensuring that **Step 40** in the Primary palette has the same perceived lightness as **Step 40** in the Surface or Error palette, making accessible contrast pairs predictable.

## 3. Semantic Mapping (Functional Interface)

Developers utilize role-based tokens to build UI. These tokens automatically resolve to the correct tonal step per theme using `light-dark()`.

### Surfaces & Backgrounds
| Semantic Token | Light Mode step | Dark Mode step | Role |
| :--- | :--- | :--- | :--- |
| `--color-surface` | surface-100 | surface-10 | Base container background |
| `--color-surface-1` | white | surface-20 | Elevated Level 1 |
| `--color-surface-4` | surface-70 | surface-40 | Elevated Level 4 (Highest) |
| `--color-on-surface` | surface-10 | surface-90 | Standard text on surfaces |

### Interactive & Actions
| Semantic Token | Light Mode step | Dark Mode step | Role |
| :--- | :--- | :--- | :--- |
| `--color-button-accent` | primary-20 | primary-70 | High-emphasis branding action |
| `--color-button-cta` | error-40 | error-60 | Destructive / Final action |
| `--color-link` | primary-30 | primary-80 | Textual navigation |

## 4. Accessibility & Contrast Matrix

Accessibility is a hard constraint for Cyan. We enforce **WCAG AA (4.5:1)** contrast for all typography and iconic content.

### Contrast Guardrail: The Elevation 4 Rule
In Dark Mode, as surfaces elevate, they become lighter to represent proximity to the light source. At **Elevation 4**, the background is a mid-dark Vibrant Navy.

> [!CAUTION]
> **Mandatory White Text:** On `surface-40`, standard secondary or low-contrast text is forbidden. You MUST use `chroma-surface-100` (White) to maintain a passing 4.6:1 contrast ratio.

## 5. Usage Guidelines: Do's and Don'ts

### Do
- **Do** use semantic tokens for all UI elements.
- **Do** use `primary` for brand-affirming actions (Save, Submit, Next).
- **Do** use `love` for social validation (Likes, Favorites) rather than `error`.

### Don't
- **Don't** use brand teal for destructive actions. Error/CTA exists for a reason.
- **Don't** override a semantic token locally (e.g., `--color-text: red`). Override the underlying Chroma instead if rebranding.
- **Don't** reference HSL values outside of the `chroma.css` file.

## 7. Testing Strategy

Manual verification of the color system involves checking the following in the Style Book:

1.  **Chroma Determinism:** Ensure all tonal steps in the Primitives table map correctly to the browser's computed HSL values.
2.  **Semantic Resolution:** Verify that theme-switching correctly swaps `color-surface` between light/dark variants.
3.  **ThemeLab Stability:** Confirm that sub-site rebranding (overriding hue variables) maintains the 3-tier layering contract without visual breakage.
